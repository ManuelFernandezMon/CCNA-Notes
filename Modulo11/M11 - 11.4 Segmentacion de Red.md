# 11.4 — Segmentacion de Red

---

## 📡 Dominios de Broadcast

> [!info] ¿Que es un dominio de broadcast?
> El conjunto de dispositivos que reciben un mensaje de broadcast. Muchos protocolos usan broadcast (ARP, DHCP).

> [!warning] Los switches PROPAGAN broadcasts
> Un switch reenvia los broadcasts por **todas** sus interfaces (excepto la de entrada). No detiene los broadcasts.

> [!success] Los routers DETIENEN broadcasts
> Un router **no propaga** broadcasts. Cada interfaz del router es un dominio de broadcast separado.

```
  DOMINIO DE BROADCAST 1        DOMINIO DE BROADCAST 2
  ┌─────────────────────┐       ┌─────────────────────┐
  │ PC1  PC2  PC3       │       │ PC4  PC5  PC6       │
  │  └────┴────┘        │       │  └────┴────┘        │
  │     [Switch] ───────┼[Router]┼─────── [Switch]    │
  └─────────────────────┘       └─────────────────────┘
  El broadcast NO cruza el router ──┘
```

---

## ⚠️ Problema: dominios de broadcast grandes

> [!danger] Demasiados hosts = demasiados broadcasts
> En una red grande, los hosts generan broadcasts excesivos que **degradan el rendimiento** de toda la red.

> [!success] Solucion: subnetear
> Dividir la red en dominios de broadcast mas pequeños.
> ```
> ANTES:  172.16.0.0/16  → 1 dominio enorme (65,534 hosts)
> DESPUES: 172.16.0.0/24 → dominio de 254 hosts
>          172.16.1.0/24 → dominio de 254 hosts
> Los broadcasts solo se propagan dentro de cada /24
> ```

---

## 🎯 Por que subnetear — 3 razones

> [!info] Beneficios del subneteo
> ```
> 📉 RENDIMIENTO → menos trafico de broadcast, menos congestion
> 🔒 SEGURIDAD   → politicas entre subredes (firewall, ACLs)
> 🛡 AISLAMIENTO → un broadcast anormal afecta solo a su subred
> ```

> [!tip] 💡 Criterios para dividir subredes
> - **Ubicacion** → un /24 por piso o por edificio
> - **Grupo o funcion** → un /24 para ventas, otro para IT
> - **Tipo de dispositivo** → un /24 para servidores, otro para usuarios
