# 14.7 — Comunicacion UDP

---

## 🪶 Bajo overhead — la ventaja de UDP

> [!success] Sin conexion = sin trafico de gestion
> UDP no establece conexion. Su encabezado es pequeño y no hay trafico adicional de administracion de la red (sin handshake, sin ACKs constantes).

---

## 🔀 Rearmado de datagramas — sin reordenar

> [!warning] UDP NO reordena
> A diferencia de TCP, UDP **no rastrea numeros de secuencia** para reordenar. Simplemente entrega los datos **en el orden en que llegaron** a la app.

```
  TCP:  reordena segun secuencia → entrega en ORDEN ORIGINAL
  UDP:  entrega tal como llega   → puede ser DESORDENADO
```

---

## 🖥 Procesos de servidor UDP

> [!info] Igual que TCP en la asignacion de puertos
> Las apps de servidor UDP usan puertos **bien conocidos o registrados**. UDP recibe un datagrama para ese puerto y lo entrega a la app correspondiente.

---

## 💻 Procesos de cliente UDP

> [!info] El cliente elige un puerto dinamico
> ```
> PASO 1: El cliente selecciona un puerto EFIMERO (dinamico) como origen
> PASO 2: Usa el puerto BIEN CONOCIDO del servidor como destino
> PASO 3: Ese MISMO par de puertos se usa en toda la transaccion
> ```

> [!example] Ejemplo — consulta DNS
> ```
> Cliente:  IP:52341 (puerto efimero)  →  DNS Server: IP:53 (puerto conocido)
> Respuesta: DNS Server: IP:53  →  Cliente: IP:52341 (mismo puerto)
> ```

---

## ⚖️ Resumen — TCP vs UDP en la practica

```
╔═══════════════════════════╦══════════════════════════════════╗
║  TCP                      ║  UDP                              ║
╠═══════════════════════════╬══════════════════════════════════╣
║  Rearma en orden ORIGINAL ║  Entrega en orden de LLEGADA     ║
║  Retransmite lo perdido   ║  NO retransmite                   ║
║  Handshake previo         ║  Sin handshake                    ║
║  Informa disponibilidad   ║  NO informa disponibilidad        ║
║  Header: 20+ bytes        ║  Header: 8 bytes                  ║
╚═══════════════════════════╩══════════════════════════════════╝
```
