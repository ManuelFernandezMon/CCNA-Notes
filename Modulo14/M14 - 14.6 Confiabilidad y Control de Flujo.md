# 14.6 — Confiabilidad y Control de Flujo

---

## 🔢 Numeros de secuencia — el orden garantizado

> [!info] Como TCP reordena datos
> Cada segmento tiene un **numero de secuencia** en su encabezado. El receptor usa estos numeros para reensamblar los datos en el **orden original**, sin importar el orden de llegada.

```
  Emisor envia:  Seg1(seq=1) Seg2(seq=1461) Seg3(seq=2921)

  Llegan asi:    Seg3, Seg1, Seg2   ← desordenados por la red

  TCP reordena:  Seg1 → Seg2 → Seg3   ← usando los numeros de secuencia
```

---

## 🔁 Perdida y retransmision de datos

> [!warning] La perdida de datos ocurre — TCP la maneja
> TCP tiene mecanismos para **retransmitir** segmentos que no fueron confirmados (sin ACK).

> [!success] SACK — Selective Acknowledgement
> Caracteristica opcional negociada en el 3-way handshake. Si ambos hosts la soportan, el receptor puede confirmar **exactamente que segmentos** recibio — incluso si hay huecos (segmentos discontinuos).
> ```
> Sin SACK: "recibi hasta el byte 1000" (todo o nada desde ahi)
> Con SACK: "recibi 1-1000 y 1500-2000, falta 1001-1499"
> ```

---

## 🌊 Control de flujo — Tamaño de ventana

> [!info] Que es el control de flujo
> La cantidad de datos que el **destino puede recibir y procesar** de forma confiable. Ajusta la velocidad de transmision para no saturar al receptor.

```
  Ventana deslizante (Sliding Window):

  El receptor le dice al emisor: "puedo aceptar X bytes mas"
  El emisor envia hasta ese limite y espera confirmacion
  Conforme el receptor confirma, la ventana "se desliza" hacia adelante
```

---

## 📏 MSS — Maximum Segment Size

> [!info] El tamaño maximo de datos por segmento
> **MSS comun con IPv4: 1,460 bytes**

> [!example] Como se calcula
> ```
> MTU de Ethernet:        1500 bytes
> − Header IPv4:            20 bytes
> − Header TCP:              20 bytes
> ─────────────────────────────────
> = MSS:                   1460 bytes
> ```

---

## 🚦 Prevencion de congestion

> [!warning] Cuando la red se congestiona
> El router sobrecargado empieza a **descartar paquetes**. TCP usa varios mecanismos, temporizadores y algoritmos para evitar y controlar la congestion — reduciendo la velocidad de envio cuando detecta perdida.
